# ADR 0001: Reproducibility strategy for NorthStar models

## Context

NorthStar's two production models are stored in S3 under names like `eta_v2_FINAL.onnx` with no registry, no dataset versioning, and no experiment tracking. A retrained model cannot be reproduced because the dataset snapshot, code version, dependency set, and random seeds used to produce it are not recorded anywhere.

## Decision

We adopt a four-layer pinning strategy using MLflow, DVC, Docker, and explicit seed management.

**Environment.** Every training job runs inside a Docker image built from a `Dockerfile` that pins the base image by digest (e.g., `python:3.11.9-slim@sha256:...`). A `requirements.lock` generated by `pip-compile` pins every transitive dependency to an exact version and hash. The image digest is stored in MLflow lineage as `base_image_digest`. No training job may run outside this image.

**Data.** All training datasets are tracked with DVC backed by S3. The dataset used for each run is identified by its DVC content-hash (`sha256`), which is stored in MLflow as `dataset_hash` and in the model registry as a required lineage field. The Feast feature-set version tag (e.g., `eta-features@v47`) is also recorded. Rolling or mutable dataset references are forbidden — every run points to an immutable snapshot.

**Code.** The MLflow run records the exact `git_sha` of the training repo at launch time. The CI pipeline rejects a training job that has unstaged changes (`git diff --exit-code`). Production promotions additionally require a clean tagged release commit. Code is never patched in-place; a new commit and a new run ID are required for any change.

**Randomness.** All random seeds are set explicitly at the top of every training script: `numpy.random.seed(SEED)`, `torch.manual_seed(SEED)`, `random.seed(SEED)`, and (where applicable) `torch.use_deterministic_algorithms(True)`. The seed value is logged to MLflow as a run parameter (`random_seed`). For GPU jobs, `CUBLAS_WORKSPACE_CONFIG=:4096:8` is set in the Docker entrypoint to enforce deterministic cuBLAS operations.

## Alternatives rejected

- **Rely on S3 timestamps for dataset identity.** S3 object timestamps are mutable and are silently overwritten by re-uploads; they provide no content guarantee. We saw this fail internally when a backfill job silently replaced a training snapshot.
- **Pin dependencies in `requirements.txt` without a lockfile.** A non-locked `requirements.txt` allows transitive dependencies to float. A `scikit-learn` patch release changed `RandomForestClassifier` default behavior in 1.3.0; a lockfile would have prevented the silent model drift.
- **Use `--no-deterministic` GPU training for speed and accept non-reproducibility.** Acceptable for initial experimentation but not for production models that must be auditable or re-trainable under compliance review.

## Consequences

- Every training run has a non-negotiable overhead: building or pulling the pinned Docker image and resolving the DVC dataset snapshot. Cold-start for a new image is approximately 5–8 minutes.
- `requirements.lock` must be regenerated and the Docker image rebuilt whenever a dependency is intentionally upgraded. This is a deliberate forcing function, not a bug.
- GPU determinism (`torch.use_deterministic_algorithms`) may cause some CUDA operations to fall back to slower CPU paths. Training wall-time is expected to increase by ~10–15% on GPU jobs.

## Revisit if

The team moves to federated or on-device training where dataset content-hashing and centralized Docker images are impractical, or if a compliance requirement mandates a formal cryptographic audit trail (at which point DVC + MLflow should be evaluated against a dedicated ML metadata store such as ML Metadata / MLMD).
